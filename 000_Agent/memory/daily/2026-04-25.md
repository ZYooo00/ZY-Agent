## 這次對話解決的技術問題

### 問題 1：gh CLI 在 Git Bash 找不到

- **症狀**：直接輸入 `gh` 指令回傳 command not found
- **原因**：Git Bash 的 PATH 沒有包含 gh 安裝路徑
- **解法**：用 `where.exe gh` 找到真實路徑，之後改用完整路徑呼叫：
  ```bash
  "/c/Program Files/GitHub CLI/gh.exe" api repos/...
  ```

### 問題 2：Git Bash 的 `ln -s` 無法建真正的 symlink

- **症狀**：`ln -s` 執行後不報錯，但 `readlink` 回傳空值，`ls -la` 顯示 `drwxr-xr-x` 而非 `lrwxrwxrwx`
- **原因**：Windows 上建 symlink 需要開發者模式或管理員權限，Git Bash 的 `ln -s` 在沒有權限時會靜默失敗（建了一個複本資料夾而非連結）
- **解法**：改用 PowerShell 的 Junction（目錄連接點，不需要管理員權限）：
  ```bash
  "/c/Windows/System32/WindowsPowerShell/v1.0/powershell.exe" -Command \
    "New-Item -ItemType Junction -Path 'C:\Users\User\.claude\skills' -Target 'E:\ZYoooClaude\000_Agent\skills'"
  ```

### 問題 3：PowerShell / gh 在 Git Bash 的 PATH 問題

- **症狀**：`powershell`、`powershell.exe`、`gh`、`gh.exe` 在 Git Bash 裡都找不到
- **原因**：Git Bash 只繼承有限的 Windows PATH
- **解法**：用完整絕對路徑：
  - PowerShell：`/c/Windows/System32/WindowsPowerShell/v1.0/powershell.exe`
  - GitHub CLI：`/c/Program Files/GitHub CLI/gh.exe`

## 這次學到的知識

### Windows symlink vs Junction 的差異

| 類型 | 指令 | 需要權限 | 跨磁碟 |
|------|------|----------|--------|
| Symbolic Link | `mklink /D` 或 PowerShell SymbolicLink | 需要管理員或開發者模式 | ✅ 可以 |
| Junction（目錄連接點） | `mklink /J` 或 PowerShell Junction | ❌ 不需要 | ❌ 同磁碟才行 |

> ZY 的情境（`C:` → `E:`）是跨磁碟，但 PowerShell `New-Item -ItemType Junction` 成功了——這是 Windows Junction 的特殊行為，實際上支援跨磁碟。Git Bash 顯示為 `lrwxrwxrwx`，可正常使用。
