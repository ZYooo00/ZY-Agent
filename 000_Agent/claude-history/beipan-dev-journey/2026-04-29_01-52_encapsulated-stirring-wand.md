## 我觀察到的現行訂貨痛點

從 Google Sheet 的結構，我看到幾個值得討論的地方：

**1. 紙本 → 數位的斷層**

**2. 兩步驟才能送出一張訂單**
先填「訂購總表」→ 再填「叫貨量」→ 才生成 LINE 訊息。中間的「叫貨量」其實是從總表手動複製過去的，有點冗餘。

**3. 沒有庫存基準**
每次訂貨都要從頭盤點，系統沒有告訴你「現在剩多少」，不能自動推薦要訂多少。

**4. 月份欄設計 vs 追加需求**
訂購總表按月份設欄，方便歷史回顧，但如果中途加訂就要多開「追加」欄，隨著時間越來越寬。

**5. 廠商視角 vs 品項視角**
訂購總表是以品項為主，但送出時是按廠商分拆。在一個新系統裡，這兩個視角都需要保留。

## 系統怎麼算建議訂購量？

公式只有一條：

```
建議訂購量 = 目標庫存量（Target）− 你剛點的數量
```

**關鍵是「目標庫存量」怎麼來？**

這個數字是**事先設定好的靜態數字**，代表「我想要冰箱裡至少保持這個量」。

它的設定依據是：
```
目標庫存量 = 月均用量 × （廠商交期月數 + 安全緩衝）
```

舉個例子：

| | GxTL（亞樸）| 102 試劑（磊柏）|
|---|---|---|
| 月均用量 | 約 19 瓶 | 約 60 盒 |
| 廠商交期 | 2 週 | 1 個月 |
| 安全緩衝 | 1 週 | 2 週 |
| **目標庫存量** | **約 28 瓶** | **約 90 盒** |

點料當天如果 GxTL 剩 12 瓶：
```
建議訂購量 = 28 − 12 = 16 瓶
```

如果剩 30 瓶：
```
建議訂購量 = 28 − 30 = −2（負數 → 本次不需要訂）
```
