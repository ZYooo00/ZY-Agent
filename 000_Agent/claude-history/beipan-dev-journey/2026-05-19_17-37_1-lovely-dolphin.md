## 我補上的 4 個新想法

**1. 數據結構要用巢狀物件，不要用字串拼接**

Gemini 說用 `Organ_胚胎室: 3, Organ_儲藏室: 2`，這是 flat key。問題是一旦品項名稱或區域名稱含有特殊字元就會爆炸，而且難以迭代。

建議改成：
```js
counts: {
  "organ": { "embryo-room": 3, "storage": 2 },
  "givf":  { "culture-area": 0 }
}
```
Phase 2 加總：`Object.values(counts["organ"]).reduce((a,b) => a+b, 0)` → 乾淨、不出錯

**2. 「已修改」第四狀態（Gemini 只有三種顏色）**

Gemini 定義了：綠色=完成、藍色=當前、灰色=未完成。但漏了第四種狀態：

**黃色邊框 = 完成但事後有修改過**

場景：ZY 把胚胎室標為完成，10 分鐘後又回去改了 Organ 的數量。這時胚胎室應該要有個視覺提示「這裡的數字改過了」，避免最後確認時忘記。在 Phase 2 也同步顯示 `edited` 標記。

**3. Smart Next 加入「鄰近優先」邏輯**

目前的 Smart Next 是按陣列順序找「第一個未完成」。但如果 ZY 剛完成的是在地圖右下角的分生室，系統卻自動跳到地圖左上角的培養區，ZY 還是得走很遠。

建議在 zone 設定加一個 `adjacent: ["zone-id-1", "zone-id-2"]` 欄位，Smart Next 邏輯改為：

```
1. 先在「鄰近區域」找未完成的
2. 鄰近都完成了，再回到 routeOrder 找
```

這樣系統推薦的下一站會更符合實際走動路線。

**4. 「twoLocation 品項的 zone 對應」是目前最緊急的資料缺口**

這個問題 Gemini 沒提到，但它是整個新架構的基礎。

現在 `products.json` 裡有 `twoLocation: true` 的品項（例如 Organ），但沒有記錄「它的第二個 zone 是哪裡」。當我們把 `zone: "string"` 改成 `zones: ["A", "B"]` 陣列時，**B 的值需要 ZY 親自填**，這份資料不能靠推斷。

這是在你提供實驗室照片時要一起確認的事。
