# 培養液管理系統規劃整理
> 整理自 AI Studio 歷史對話（2026-04-27），供後續與 Claude Code 繼續討論使用。

## 一、建立系統的期許：系統需包含的條件、規則與功能

### 核心目標
以實驗室培養液（Medium）試劑的「進貨 → QC → 庫存 → 每日估備盤 → 消耗」為主軸，建立一套完整的閉環管理系統。

### 功能模組清單

#### 1. 品項主檔（Products）
- 廠商分群
- 品項名稱、類別分類（見下方類別表）
- 單位名稱（瓶/組）、單瓶容量 ml、安全庫存量（瓶）
- 開封後效期天數（如 HTF = 7 天）
- QC 所需天數（部分品項 QC 需 7 天，部分 1 天）
- 各培養液的備盤換算係數（每台 OPU/ICSI/備盤的 ml 係數）

**品項類別對照表：**

| 類別 | 品項（現有） | 備注 |
|---|---|---|
| 卵子培養液 | G-IVF、HTF（+10%SPS） | 可替換、可新增廠牌 |
| 卵子添加劑 | AOA（弘優）、AOA（508） | |
| 胚胎培養液 | GxTL、H5GT、Geri ONE | 可替換、可新增廠牌 |
| 植入培養液 | Glue | |
| Oil | Heavy Oil 等 | 各盤覆蓋用，另需估算分裝瓶數 |
| Flush medium | HEPES | 分裝罐，依 OPU 台數估算 |

---

#### 2. 訂貨系統
- 廠商分群，選擇品項與訂購數量（瓶/組）
- 記錄訂購日期、訂購人
- 訂單狀態：下單中 / 待收貨 / 已入庫 / 取消
- 顯示目前庫存快照（訂貨當下核對用）

#### 3. 收貨/入庫系統
- 對應已下訂的訂單，顯示待收貨清單
- 收貨時填入：實際到貨量、廠商批號、廠商效期、入庫日期
- 支援分批到貨（部分到貨 / 結案邏輯）
- 入庫後自動在庫存批號表新增一筆（Automation），QC 狀態預設「待測試」
- 效期檢核：新收批次效期應比舊批次更晚（如不符則跳出提醒）

#### 4. QC 管理
- 顯示所有「待測試」批次清單
- 顯示「上一批預計何時用完」（以平均日消耗量推算）
- 顯示「最晚 QC 截止日」= 上一批用完日 - QC 所需天數
- QC 通過後，批次才能進入備盤選單（防呆機制）

#### 5. 估盤系統（每日規劃）
- 操作人員：抓盤人 2 位
- 依「選擇當天使用的培養液」動態顯示對應盤數輸入欄位
- **卵子培養液專區**（Show_If: 類別=卵子培養液）
  - OPU 台數、De 盤數、TE 盤數、AOA 盤數、精蟲(mL)、多配_卵子(mL)
  - 公式：`OPU*3 + (De+TE+AOA)*0.5 + 精蟲 + 多配`
- **卵子添加劑專區**（Show_If: 類別=卵子添加劑）
  - AOA 盤數
  - 公式：`AOA * 0.3`
- **胚胎培養液專區**（Show_If: 類別=胚胎培養液）
  - D0、D3、D3+Bx、Bx、TB、Geri、pH dish 盤數、多配_胚胎(mL)
  - 公式：`(D0+D3+Bx+TB+Geri+pH_dish)*0.5 + 多配`
  - 注意：同一天可能用多種胚胎培養液，各自填寫盤數
- **植入培養液專區**（Show_If: 類別=植入培養液）
  - Glue 盤數
  - 公式：`Glue * 0.8`
- **Oil 專區**（Show_If: 類別=Oil）
  - 輸入：總受精數、STD 台數、Spindle 盤數、Bx 盤數
  - 系統估算公式：`CEILING(((總受精數/3)*7 + (Spindle+Bx)*5 + STD*10 + 50) / 50) * 50` ml
  - Oil 以 50ml 分裝瓶計算
- **Flush medium 專區**（Show_If: 類別=Flush medium）
  - 輸入：OPU ≤5 幾台、OPU 6-9 幾台、OPU ≥10 幾台、額外指定（梁+金幾台）
  - 系統估算公式：`MAX(8, CEILING(OPU_小5*0.5 + OPU_69*1 + OPU_大10*2 + 額外 + 1))`
  - HEPES 以分裝罐計，結果為需準備幾罐
- 系統自動顯示「估算消耗量」（唯讀）
- 此階段**不需要**確認批號與是否開封

#### 6. 備盤系統（每日執行）
- 操作人員：備盤人 2 位
- 顯示今日估盤結果（唯讀，包含各培養液的建議用量）
- 備盤人逐項核對，依實際情況填寫：
  - 指定批號（過濾 QC 通過 + 有庫存的選項）
  - 實際使用量（預設等於估算量，可手動微調）
  - 新開瓶數（填 1 或更多，影響庫存扣除）
  - 若新開瓶數 > 0，自動帶入今日作為開封日期
- 效期防呆：若所選批號「備盤日 + 1 天」後已過開封效期，禁止送出
- 完成備盤後標記 ✅，支援逐項 Check 介面
- 備盤完成後**才實際觸發庫存扣除**

#### 7. 庫存批號（Inventory）
- 每批貨對應一筆紀錄（一批 = 60 瓶 → 只建 1 筆，欄位記數量）
- 欄位：批號 ID、品項 ID（連品項主檔）、廠商批號、廠商效期、初始數量、目前剩餘量（系統計算）、QC 狀態、開封日期（備盤時回寫）
- 目前剩餘量 = 初始數量 × 單瓶容量 - 累計已用量（Virtual Column）

#### 8. 低庫存警示
- 計算目前剩餘瓶數 = 總剩餘 ml / 單瓶容量
- 若剩餘瓶數 ≤ 安全庫存量，品項名稱變紅 + ⚠️ 提示

#### 9. 報表後台
- 每個批號的開始使用日 / 最後使用日（MIN/MAX）
- 每月各品項使用量趨勢圖（Chart）
- 各品項庫存現況總覽

## 二、優化溝通流程：未來討論時可調整的方式

### 問題分析
本次 AI Studio 對話歷時很長，主要原因：
1. **工具選擇反覆**：從 Airtable 切換到 AppSheet 花了大量篇幅
2. **邊學邊調整**：沒有先確定架構，邊做邊發現邏輯問題
3. **截圖 debug**：大量截圖 + 文字來回確認，成本高
4. **AI 的假設錯誤**：多次出現「公式裡用了沒有的欄位名稱」、「欄位類型設錯」等問題

### 未來優化建議

**在開始前先做：**
- 確定工具選擇（不要中途換）→ 本次最終決定：**AppSheet**
- 先在紙上或文件確認好「資料表結構（欄位名稱、類型、關聯）」再動手
- 先建好 Google Sheet 骨架，再進入 AppSheet 設定

**對話時：**
- 問問題時分清楚「這是邏輯問題」還是「這是操作問題」
- AI 給步驟時，先確認理解再執行，不要一次做太多步
- 發現錯誤時，截圖時附上「我做了哪些步驟」一起說，避免猜測

**工具使用：**
- AppSheet 最常踩的坑：欄位 Type 要從 Text 改成正確類型（Number/Date/Ref/Enum）
- 每次改 Google Sheet 後，都要在 AppSheet 按 Regenerate Structure
- Key 欄位的 Initial Value 務必設 `UNIQUEID()`，不能用 RowNumber

## 三、達成結果的方法：考慮用 Vibe Coding 做小頁面

### 現況評估
目前 AppSheet 的進度：
- Google Sheet 的 5 張表架構已規劃完成（待實作）
- AppSheet 訂貨/收貨的 Automation 設定卡關（Action 的 "For a record of this table" 設錯位置）
- 估盤/備盤系統尚未開始建

### 使用 Vibe Coding 的可能方向

捨棄 Airtable / AppSheet 的限制，直接用程式做一個客製化小頁面：

**優勢：**
- 完全自訂介面與邏輯，不受工具限制
- 可以直接做成類似 Google Form 的簡單頁面，放在公司內網使用
- 估盤/備盤的複雜公式（Oil、HEPES）直接用 JavaScript 寫，沒有「公式不支援」的問題
- 資料存在 Google Sheets 作為後端（用 Google Apps Script API），不用換資料庫

**建議技術方向：**
- 前端：純 HTML + Tailwind CSS（簡單易懂）
- 資料後端：Google Sheets + Apps Script Web App（免費、已熟悉）
- 部署：Google Apps Script 直接 Deploy 成 Web App，公司帳號可存取

**可先做的最小版本（MVP）：**
1. 訂貨表單 → 寫入 Google Sheet
2. 收貨確認 → 觸發入庫（Apps Script）
3. 估盤頁面 → 填台數/盤數 → 算出各培養液需求量
4. 備盤登記 → 確認批號、填實際用量、觸發庫存扣除

## 四、文件審核與規範：最終需要整理的文件

> 此部分待與 ZY 逐項討論後補齊內容。以下為架構草稿。

### (a) 東西進入實驗室的流程

**訂貨流程：**
- 盤點現有庫存（與系統數量核對）
- 每月/每兩週估算各品項需求量
- 填寫訂購單（廠商、品項、數量）→ 送出

**收貨流程：**
- 貨到確認訂購單
- 確認到貨量是否與訂購量相符（全到 / 部分到貨）
- 登記批號、廠商效期（需比現有庫存效期晚）
- 批次自動進入庫存，QC 狀態設「待測試」

**QC 流程：**
- 新批號入庫後依品項 QC 所需天數安排測試
- 確認最晚 QC 截止日（避免斷糧）
- 測試通過 → 狀態改為「通過」，批號進入可使用清單

---

### (b) 所有的盤點與整合

**庫存盤點：**
- 定期核對系統剩餘量 vs 實際冰箱數量
- 若有差異，手動修正並記錄原因

**品項主檔整合：**
- 統一維護廠商資訊、品項規格、換算係數
- 新增/停用品項時更新品項主檔
- 培養液分類（類別欄位）是所有 Show_If 邏輯的核心，需保持準確

**批號整合：**
- 同品項可能同時存在多批號，以效期先進先出為原則
- 開封後效期（7 天、28 天等）由系統自動管理

---

### (c) 物品使用時的相關規範

**開封與使用規範：**
- 使用前確認批號是否通過 QC
- 開封日期須記錄，系統自動計算推算過期日
- 備盤日當天或次日使用的批號，其推算過期日必須 ≥ 備盤日 + 1 天（系統防呆）
- 每次開封新瓶，系統記錄開封瓶數，自動從庫存扣除

**備盤換算規則（最終確認版）：**

| 培養液 | 使用盤子 | 計算公式 |
|---|---|---|
| 卵子培養液（G-IVF / HTF） | OPU、De、TE、AOA、精蟲(mL)、多配(mL) | `OPU*3 + (De+TE+AOA)*0.5 + 精蟲 + 多配` |
| 卵子添加劑（AOA） | AOA | `AOA * 0.3` |
| 胚胎培養液（GxTL / H5GT...） | D0、D3、D3+Bx、Bx、TB、Geri、pH dish、多配(mL) | `(各盤加總) * 0.5 + 多配` |
| 植入培養液（Glue） | Glue | `Glue * 0.8` |
| Oil | 分裝 50ml 瓶 | `CEILING(((受精/3)*7 + (Spindle+Bx)*5 + STD*10 + 50) / 50) * 50` |
| Flush medium（HEPES） | 分裝罐 | `MAX(8, CEILING(≤5台*0.5 + 6-9台*1 + ≥10台*2 + 額外 + 1))` |

**報廢規範：**
- 廠商效期到達：整批報廢
- 開封後超過效期天數（如 7 天）：該瓶報廢，系統記錄
- 報廢時記錄原因，不計入正常消耗
