## 一、平台分工

| 平台 | 使用情境 | 負責功能 |
|---|---|---|

| **電腦 Web** | 固定在工作站螢幕旁 | 備盤、進貨記錄、庫存總覽、訂貨管理、後台設定 |

## 二、手機 App 需求

### 2.1 導航架構

底部 Tab（2 個主 Tab）：
- 📋 今日估盤
- 📦 庫存盤點

### 2.2 估盤頁

**台數輸入**：每個台類各一個文字輸入框，點擊彈出手機數字鍵盤。

台類清單：OPU / De / TE / AOA / 精蟲

**自動計算邏輯**：台數輸入後，系統即時顯示：
- 各盤子建議數量
- 各培養液建議用量（mL）
- HE / Oil 估算結果

**估盤流程**：
```
估盤人甲
  └─ 輸入各台類台數
  └─ 系統自動帶出建議盤數 + 用量
  └─ 可手動調整任何欄位
  └─ 點「送出估盤」→ 系統記錄 + 通知估盤人乙核對

估盤人乙（核對）
  └─ 開啟今日估盤單
  └─ 確認無誤 → 按「核對確認」→ 記錄確認人 + 時間戳
  └─ 有異動 → 修改後通知估盤人甲再確認
```

**特殊規則**：
- Spindle 台數 = De 台數，系統自動帶入，不需另外輸入
- OPU 多備規則：系統附註參考標準（AMH / NS E2 / T2O），實際由胚胎師手動決定
- pH 測試盤提醒：估盤頁顯示各培養箱距上次滅菌天數，詢問是否安排 pH 盤

**估盤確認後的鎖定規則（Plan vs Actual）**：
```
估盤確認 = 計畫版 Plan（保留不動，作為稽核基準）
備盤完成 = 實際版 Actual（允許臨時修改）
差異記錄 = 估盤 X 個 → 實際備 Y 個，備盤人填原因備註
```

**估盤 → 備盤資料橋接規格（localStorage，Session 3 實作）**：

> 手機版 gupan.html 確認後，透過 `localStorage` 將估盤結果傳送給電腦版 beipan.html。

```
Key：gupan-confirmed
寫入時機：估盤人乙按「核對確認」（State C）
讀取時機：beipan.html 頁面 init()

資料結構：{
  date: "YYYY-MM-DD",
  confirmedAt: "HH:MM",
  staffA: string,         // 估盤人甲姓名
  staffB: string,         // 估盤人乙姓名
  taiVals: { opu, te, szu, fbt, bt, tbx },
  givf: {                 // 索引對應 givfDishes[0..7]
    opu, de, te,          // 0=OPU 1=De 2=TE
    aoa_ci, aoa508,       // 3=AOA弘優 4=AOA508
    sperm,                // 5=精蟲（台數）
    givf_ph,              // 6=G-IVF pH
    givf_x                // 7=多配（mL）
  },
  matrix: {               // 各盤子各有 h5（H5GT欄）和 gx（GxTL欄）
    geri_szu: { h5, gx }, d0: { h5, gx },
    bx: { h5, gx },      tb: { h5, gx },
    glue: { h5, gx },    geri_fbt: { h5, gx },
    ph: { h5, gx },      multi: { h5, gx }
  }
}
```

**「今日需用」即時計算邏輯（recalcAllNeeded）**：

任何 GIVF 表格、胚胎矩陣、Glue 卡片的 `oninput` 事件，均觸發完整重算：

| 培養液 | 公式 |
|---|---|
| G-IVF | OPU×3 + (De+TE+AOA弘優+AOA508)×0.5 + 精蟲台×0.7 + G-IVF pH×1 + 多配mL |
| AOA CI（弘優）| AOA弘優盤數 × **0.3 mL** |
| AOA GM508（明美）| AOA508盤數 × **0.3 mL** |
| H5GT | (D0+Bx+TB+Geri-受精+Geri-FBT)×0.4 + H5GT pH×1 + 多配mL |
| GxTL | (D0+Bx+TB+Geri-受精+Geri-FBT)×0.4 + GxTL pH×1 + 多配mL |
| EmbryoGlue | Glue盤數 × **0.8 mL** |

### 2.3 庫存盤點頁

- 以存放位置分組（冰箱位置 / 常溫櫃）
- 逐品項輸入當前看到的實際數量
- 每個品項有**效期備註欄**（選填），可記錄最後一批的效期與剩餘量（例：「06/30 剩 1 瓶」）
- 效期備註存入 `pandian-result.allValues[n].note`，電腦版訂貨頁的培養液 Section 可直接顯示
- 完成後系統比對安全庫存量，標示需補貨品項

## 三、電腦 Web 需求

### 3.1 導航架構

左側或頂部導航欄，功能模組：
- 今日備盤
- 進貨記錄
- 庫存總覽

### 3.2 備盤頁

**進入條件**：估盤已確認後才解鎖當日備盤頁。

**批號選擇介面（列表式）**：

```
試劑     批號      開封日  到期日  剩餘    今日需用  狀態
GIVF    53189    4/25   5/1   3.2mL   5.0mL  ⚠️ 不足
GIVF    53651    4/28   5/4   47mL    5.0mL  ✅ 充足
GxTL    44201    4/27   5/3   38mL    12mL   ✅ 充足
H5GT    88392    4/25   5/1   6mL     12mL   🔴 1天
GIVF    52100    —      —     —       —      ❌ 已過期（不可選）
```

**效期顏色系統**：

| 狀態 | 顏色 | 條件 | 操作 |
|---|---|---|---|
| 充足 | 🟢 綠色 | 距到期 > 橙色門檻 | 可選，FIFO 優先推薦最早開封 |
| 緊迫 | 🟠 橙色 | 距到期 ≤ 橙色門檻（預設 3 天）| 可選，顯示警示文字 |
| 危急 | 🔴 紅色 | 距到期 ≤ 1 天 | 可選，顯示「明天到期，今日最後機會」|
| 封鎖 | ⬛ 灰色 | 當天或已過期 | 無法選入備盤 |

**試劑耗盡三種情境**：

| 耗盡方式 | 系統處理 |
|---|---|
| 正常用完 | 備盤人點「此批已用完」，記錄用完日期 |
| 開封天數到期 | 系統自動標記「已過效期」，備盤時無法選擇 |
| 原廠效期到期 | 收貨時輸入原廠到期日，超過時禁止選入備盤 |

**備盤流程**：
```
備盤人甲
  └─ 查看已確認估盤單
  └─ 選擇各試劑使用批次
  └─ 若需開新瓶 → 登記開封，系統自動記錄開封日，效期計時開始
  └─ 填入實際備盤數量（可與估盤不同，需填原因備註）
  └─ 備盤完成後觸發庫存扣除

備盤人乙（點盤）
  └─ 核對盤數（只核對數量，批號/開封日由備盤人甲負責）
  └─ 確認 → 記錄確認人 + 時間戳
```

**GM508 特殊規則**：
- 備盤時出現「客戶病歷號」必填欄位
- 記錄格式：批號 × 使用日期 × 病歷號

**備盤 UX 細節**：

| 項目 | 決定 |
|---|---|
| 批號選擇時即時顯示剩餘量 + 今日是否足夠 | ✅ V1 納入 |
| 備盤草稿自動儲存（防意外中斷）| ✅ V1 納入 |
| GM508 備盤時必填客戶病歷號 | ✅ V1 納入 |
| Spindle 台數 = De 台數，系統自動帶入 | ✅ V1 納入 |
| 多配欄位顯示系統建議值，可手動覆蓋 | ✅ V1 納入 |
| 多配 mL 調整 → 右欄「今日需用」即時更新 | ✅ V1 納入（prototype 已驗證）|
| 多批同天各自記錄用量（非拼接格）| 🔜 UI/UX 細節討論時確認 |
| 並發問題（兩人同時備不同盤）| ⬜ 低風險，V1 不特別處理 |
| AOA 盤分拆：AOA 弘優盤 + AOA 508 盤各自輸入 | ✅ V1 納入（prototype 已驗證）|

**批號狀態三態邏輯（2026-04-30 確認）**：

| 狀態 | 條件 | 顯示 |
|---|---|---|
| ✅ 充足 | 已開封剩餘 ≥ 今日需用量 | 綠色 badge |
| 🔄 需開新瓶 | 開封瓶不足，但同批次總量（含未開封）≥ 今日需用量 | 藍色 badge |
| ⚠️ 批號不足 | 同批次總量 < 今日需用量 | 橘色 badge，自動展開第二批號面板 |

> 可用量計算：`已開封剩餘 mL + 未開封瓶數 × 單瓶容量`

**批號不足時的混批流程**：
- 選到「批號不足」→ 自動展開第二批號面板 + 顯示黃色 banner
- 第二批號面板底部有「混批備註（必填）」欄位
- 選回充足批號 → banner 收合

**登記新開瓶**：
- 點選「登記新開瓶」→ 選擇批號 → 確認登記
- 系統即時更新：未開封瓶數 −1，已開封剩餘 + 單瓶容量
- 批號狀態 badge 自動重算

**Oil / HEPES 備盤記錄（2026-04-30 新增）**：
- 左欄新增獨立卡片，顯示「待分裝（估盤建議）罐數」和「今日開封瓶數」
- 今日開封瓶數不與估盤掛鉤，僅用於庫存扣除記錄
- 今日批號欄位（選填）

### 3.3 進貨記錄頁

> 本節需求於 2026-05-05 大幅更新，整合掃碼自動識別、訂單自動比對、批次暫存流程。

**設計原則**：手機優先。電腦開啟手機版頁面可用（畫面較寬），不另做獨立電腦版。

---

#### 3.3.1 完整收貨流程（掃碼 → 暫存 → 入庫）

```
掃描包裝條碼
  ↓
系統自動識別品項（GTIN 對照 品項條碼資料庫）
  └─ 識別成功 → 顯示品項卡片（品名 + 廠商 + LOT + 效期）
  └─ 識別失敗 → 提示「條碼未建檔」，允許手動選擇品項繼續
  ↓
自動比對待到貨訂單
  └─ Path A（有訂購）→ 顯示訂單資訊：訂購量 | 已到量 | 待到貨量
       └─ 多筆 pending → 列清單讓使用者點選（時間倒序）
  └─ Path B（無訂購）→ 橘色警示「查無對應訂單，請確認是未登記還是多收」
  ↓
加入本次暫存區（session tray）
  ↓
重複掃下一個品項
  ↓
點「審核本次進貨」
  ↓
批次必填（簽收人 + 到貨日期，套用至全部品項）
  ↓
確認無誤 → 批次新增 → 正式寫入進貨記錄
```

---

#### 3.3.2 品項識別資料來源

- GTIN 對照表：`品項條碼資料庫.md`（固定欄位：盤點名稱、正式品名、廠商、REF、GS1 (01)、規格）
- LOT / 效期每批不同，由掃碼當下的 GS1 條碼解析（AI 10 = 批號、AI 17 = 效期 YYMMDD）
- 無 GS1 的品項（如 Geri filter、Mounting Glue）：僅能手動輸入，不支援掃碼識別

---

#### 3.3.3 session tray（本次暫存區）

- 記憶體陣列，不寫 localStorage，頁面重整即清空
- 每次掃完確認後加入 tray，不立即入庫
- tray 顯示於畫面下方「本次進貨紀錄」卡片（可展開 / 收合）
- 每筆支援：單筆刪除 + 點擊進入編輯

---

#### 3.3.4 警示分層設計

| 警示類型 | 觸發時機 | 呈現方式 |
|---------|---------|---------|
| 無對應訂單 | 掃碼識別後立即顯示 | 橘色提示卡，可繼續；贈品 / 試用品照此路徑加入 |
| 效期異常（過期 / 比現有舊）| 加入 tray 時 + 審核頁 | 每筆 badge 標記，且審核頁彙整列出需確認 |
| 超量到貨（qty > 訂購量）| 審核頁批次確認前 | 紅色警示，**必填超量原因備注**才能繼續 |
| 數量不足（qty < 訂購量）| 審核頁批次確認前 | 橘色提示，允許繼續（可能分批到貨）|
| 後續需要行動（如 102 QC）| 批次新增完成後 | in-app 跳出提示；LINE / Email 通知為未來整合項目 |
| 品質 / 包裝異常（凹損、溫控）| 使用者手動觸發 | 每筆旁小型低調連結 → 文字備注；照片上傳為後續功能 |

警示規則細節（效期至少 N 天、數量差 X% 算超量）待後續確認後設定。

#### 3.3.5 拒收 / 退貨記錄

- 收貨後發現問題選擇退貨：建立一筆狀態為「拒收」的紀錄，到貨數量 = 0，備注填退貨原因

#### 3.3.7 與訂單系統的串接（2026-05-11 更新：order-history 架構）

訂單資料現在以 **`order-history` 陣列**（localStorage）儲存，每筆訂單為獨立物件，取代原本的單一 `order-result`：

- 每筆到貨記錄透過 `orderId`（= gupanId，如 `f-102`）連結至 `order-history` 中的對應訂單
- 到貨確認後呼叫 `updateOrderStatus(orderId, receivedQty)`，自動更新對應訂單的 `receivedQty`，重新計算 `pendingQty`
- 顯示格式：「訂購 60 ｜ 已收 20 ｜ **待到貨 40**」
- 全數到貨後訂單狀態自動更新為 `closed`，從履約中心消失
- 欠貨 / 分批到貨：訂單狀態改為 `partial`，保留於履約中心持續追蹤

**FIFO 到貨配對邏輯**：
- `updateOrderStatus()` 從最舊訂單（陣列末端）開始配對，逐筆扣減 pendingQty
- `rollbackOrderQty()` 從最新訂單（陣列開頭）回溯（LIFO），用於作廢進貨記錄

**compat shim**：`loadIntransit()` 同時彙整所有 pending/partial 訂單寫入 `order-result`，確保 kucun.html KPI「待收貨」欄持續正常運作，不需修改 kucun.html。

### 3.3.8 履約中心 — 待到貨橫向卡片區（2026-05-11 新增）

> 位於 jinhuo.html（進貨記錄頁）頂端的橫向可滾動卡片區，讓使用者快速掌握所有在途訂單並一鍵快速收貨。

**觸發條件**：`order-history` 中有任何 `pending` 或 `partial` 狀態的訂單時出現；無欠貨時自動隱藏。

**UI 設計**：
- 橫向可滾動（`overflow-x:auto`，iOS/Android Samsung 均支援慣性滑動）
- 按廠商分組，每個廠商一張卡片
- 卡片內列出該廠商所有欠貨品項，每行顯示「品名 欠 N 瓶」+ [📥 快收] 按鈕

**廠商狀態徽章**：
- 藍色「待到貨」：該廠商所有品項均為 pending（尚未到貨）
- 橙色「部分到貨」：該廠商有任何品項已部分到貨（partial）

**📥 快收（Quick Receive）流程**：
1. 點擊 [📥 快收] → 掃描 Modal 從底部滑出，直接進入 Step 2（跳過相機）
2. 品項卡自動識別（綠色品項卡），訂單資訊自動填入（藍色訂單區）
3. 預設數量 = 該品項的 pendingQty
4. 使用者只需補填批號 + 效期 → 點「加入暫存」
5. 若需掃條碼 → 點「重新掃」可啟動相機（不影響原有流程）

### 3.5 訂貨管理頁

> 根據最新庫存盤點結果，決定本次各品項訂購數量，並追蹤訂單到貨狀態。

**頁面上方：盤點資訊卡片**
- 顯示最新一次盤點日期、盤點人、低於安全庫存警戒線的品項數

### 4.3 HEPES / Oil 估算公式

> 2026-04-30 更新：HE 正式名稱為「HEPES」，無 HIPPS 一詞。

**HEPES（HE）：**
```
HEPES 罐數 = MAX(8, ROUNDUP(≦5台×0.5 + 6-9台×1 + ≧10台×2 + 梁+金台數 + 卵多的賴台數 + 1, 0))
最低保底 8 罐
```

**Oil（= 覆蓋油）：**
```
Oil 份數 = ROUNDUP(((總受精數÷3)×7 + (Spindle盤+Bx盤)×5 + STD台×10 + 50) ÷ 50, 0)
```
輸入欄位：總受精數、STD 台數、Spindle 盤數、Bx 盤數

### 4.4 庫存計算邏輯

```
批號剩餘量（mL）= 初始數量 × 單瓶容量 − 累計已用量
```

- 庫存扣除時間點：備盤完成確認後，而非選批號當下
- 兩種效期並存（備盤選批時同時過濾）：
  - 原廠效期：收貨時記錄，超過時禁止選入備盤
  - 開封後效期：備盤時記錄開封日，超過時禁止選入備盤

### 4.5 order-history 訂單歷史資料結構（2026-05-11 新增）

儲存於 `localStorage['order-history']`，格式為 JSON 陣列（newest-first）：

```json
[
  {
    "id": "ord_1715400000000",
    "confirmedAt": "2026-05-11T08:00:00.000Z",
    "status": "pending",
    "orders": [
      {
        "orderId": "f-102",
        "name": "102（磊柏）",
        "vendor": "磊柏",
        "unit": "盒",
        "orderQty": 60,
        "receivedQty": 20,
        "pendingQty": 40
      }
    ]
  }
]
```

**欄位說明**：

| 欄位 | 說明 |
|------|------|
| `id` | 訂單 ID（`ord_` + timestamp）|
| `confirmedAt` | 訂單送出時間（ISO 8601）|
| `status` | `pending`（待到貨）/ `partial`（部分到貨）/ `closed`（全數到貨）|
| `orders[].orderId` | = gupanId（如 `f-102`），用於 `GUPAN_MAP` 反查 productId |
| `orders[].orderQty` | 本次訂購數量 |
| `orders[].receivedQty` | 已到貨數量（累計）|
| `orders[].pendingQty` | 尚待到貨 = orderQty − receivedQty |

**狀態轉換**：
```
pending → partial（有任何品項部分到貨）
partial → closed（所有品項 pendingQty = 0）
pending → closed（一次全到）
```

**compat shim**：`loadIntransit()` 讀取 `order-history` 後，彙整所有 pending/partial 訂單聚合為單一 `order-result` 物件寫回 localStorage，確保 kucun.html 的「待收貨」KPI 不受影響。

## 五、進貨 × 備盤完整串聯鏈

```
進貨收貨（批號 + 廠商效期 + 到貨量）
  ↓
QC 管理（目前僅 102 需 QC，其餘免）
  ↓
估盤（早上，抓盤人 2 位）
  └─ 輸入台數 → 系統算建議用量 → 第二人確認
  ↓
備盤（下午，估盤確認後才解鎖，備盤人 2 位）
  └─ 選批號（過濾：QC 通過 + 未超原廠效期 + 未超開封效期）
  └─ 填新開瓶數 → 系統自動記錄開封日，效期計時開始
  └─ 備盤完成後才觸發庫存扣除
  ↓
庫存自動更新（剩餘量 = 初始 mL − 累計已用 mL）
  ↓
低庫存警示（剩餘瓶數 ≤ 安全庫存量 → 提醒訂貨 → 回到進貨流程）
```
